## 3. Documented adaptations from plan-as-written

### 3.1 ESM `__dirname` substitution in `scripts/build-index.ts` and `scripts/copy-index.ts`

**Trigger:** `package.json` has `"type": "module"`. The plan's verbatim CLI-entry guards use `__dirname` (CJS-only).

**Fix:**
- `build-index.ts`: substituted `path.resolve(new URL(import.meta.url).pathname, ...)`.
- `copy-index.ts`: substituted `path.dirname(fileURLToPath(import.meta.url))`.

**Status:** accepted. Necessary for ESM compatibility.

### 3.2 LANDED 4-arg `BotClient.sendMessage` signature in `/help`, `/search`, `/ask`

**Trigger:** plan's command tests + impls used a simplified 3-arg `sendMessage(chatId, text, opts?)` form that does not match the LANDED `BotClient` interface from E1's `src/telegram/client.ts` (which exports the 4-arg `sendMessage(chatId, threadId, text, parseMode?)`).

**Fix applied uniformly across `commands/{help,search,ask}.ts` and their tests:**
- All test mock-call indexing shifted: `mock.calls[0]?.[1]` → `mock.calls[0]?.[2]` for the text arg.
- All impl `sendMessage` calls became 4-arg: `bot.sendMessage(chatId, parsed.topicId ?? undefined, text, parseMode?)`.
- The 4th positional `parseMode` is the literal `"MarkdownV2"` (not `{ parse_mode: "MarkdownV2" }` as plan's verbatim suggested).
- `parsed.topicId ?? undefined` translates `null` (ParsedMessage.topicId type) to `undefined` (BotClient signature accepts `number | undefined`).

**Status:** accepted. Matches the E1 closeout's documented BotClient adaptation pattern.

### 3.3 Webhook-route dispatch placement: above DM/supergroup split, not "mirror /gbrain-forget"

**Trigger:** spec §4 says `/help`, `/search`, `/ask` work in BOTH DM and topic surfaces; plan's wiring text said "Mirroring the existing `/gbrain-forget` branch" but that branch is DM-only.

**Fix:** Inserted a new dispatch block ABOVE the `if (isDM)` line in `webhook/route.ts`. It first constructs `const parsed = parseMessage({ raw: msg, topics: buildTopicMap(cfg) })` once, then the three `if (isCommand(text, "/...")) { ... return handle...({ parsed, config: cfg, bot }); }` branches in order: `/help`, `/search`, `/ask`. The existing DM-only commands (`/gbrain-*`, `/yes`/`/no`) and the supergroup ingest pipeline are unchanged. `ingestOne` continues to call `parseMessage` internally — minor parse-twice overhead, acceptable for E2; refactor punted to a future cleanup.

**Status:** accepted. Spec-faithful interpretation.

### 3.4 `tests/safety/injection-cases.test.ts` (with `.test.ts` suffix), not the plan's bare `.ts`

**Trigger:** Vitest 3.x intersects positional file arguments with the configured `include: ["tests/**/*.test.ts"]` glob. The plan's bare-`.ts` filename is unreachable even via `pnpm test tests/safety/injection-cases.ts` — Vitest reports "No test files found".

**Fix:** Renamed the file to `injection-cases.test.ts` so it auto-discovers. Live tests remain gated by `describe.runIf(HAS_KEY)` (skipped when `GEMINI_API_KEY` is unset, so no quota burn in CI). The pure citation-validator `describe` block at the bottom now runs as part of the default test suite — duplicating coverage already in `tests/retrieval/cite.test.ts` but harmless.

**Status:** accepted. Vitest 3 behavior change forced this.

### 3.5 `/ask` test asserts `toContain("\\[1\\]")`, not `"[1]"`

**Trigger:** the plan's impl correctly escapes `[` and `]` for MarkdownV2 (`\\[${id}\\]` → string `\[1\]`); the plan's test asserted unescaped `"[1]"` substring, which would not match the escaped form.

**Fix:** test asserts `text.toContain("\\[1\\]")` (which in JS source is the literal string `\[1\]`).

**Status:** accepted. Test-assertion correction.

### 3.6 Defensive narrowing for strict-TS settings

**Trigger:** `tsconfig.json`'s `noUncheckedIndexedAccess` flags various tuple/array index accesses as `T | undefined`.

**Fix:** added narrowing fallbacks in three places:
- `scripts/build-index.ts:179`: `process.argv[1] ?? ""` (CLI guard).
- `src/commands/ask.ts:59,60`: `r.entry.source_lines[0] ?? 0`, `[1] ?? 0` (schema-guaranteed but TS doesn't narrow tuples).
- `src/commands/{search,ask}.ts`: `rl.retryAtIso ?? "later"` for the optional `retryAtIso` field on the `RateLimitResult` failure branch (defensive; the field is set in practice but the type allows undefined).

**Status:** accepted. All are narrow safety adaptations.

## 5. Architectural decisions logged in E2

- **Public-command dispatch placement:** `/help`, `/search`, `/ask` dispatch lives ABOVE the DM/supergroup split. The route's `parseMessage(...)` call is now run on every webhook hit (was: only in supergroup ingest). Slight overhead (~one regex pass + Map lookup per hit) — accepted in exchange for the cleaner uniform shape across DM and topic surfaces.
- **Fixture archive layout:** `tests/fixtures/archive/` mirrors the production `community/archive/` layout (`2026-04/`, `digests/`). The build-index integration test copies it into a `tmpRepo/community/archive/` for hermetic runs. Fixture content is deliberate — the poisoned fixture's `AIzaSyBexampleNotARealKeyABCDEFGHIJ123456` (37 chars after `AIza`, 35-char regex match) is the seed for the Cat C sensitive-data extraction test.
- **Vitest 3 file-discovery contract:** `tests/safety/injection-cases.test.ts` uses the standard `.test.ts` suffix and relies on `describe.runIf(HAS_KEY)` for live-test gating. The plan's bare-`.ts` filename was incompatible with Vitest 3's positional-arg + include-glob intersection.
- **MarkdownV2 escaping discipline at the command boundary:** `commands/search.ts` and `commands/ask.ts` apply `escapeMd()` to all user-content (author handle, topic, date, content_preview, query, validatedAnswer). Plain-text replies (usage, rate-limit message, index-unavailable) intentionally use the default "Markdown" parse mode for simpler text — they contain no MD-special chars.
- **Citation marker fidelity:** `validateAndPruneCitations` runs BEFORE `escapeMd`. The validated answer (with `<citation id="N"/>` tags intact for valid IDs) is then escapeMd'd, which escapes `>` → `\>` and `.` → `\.` in MarkdownV2. The user sees the raw XML tag in the answer body — matches spec §4.1 step 11's `{answer text with citation markers}` design.
- **CLI entry-point detection:** both build scripts use `import.meta.url === \`file://${process.argv[1] ?? ""}\`` to detect direct invocation vs. import. The `?? ""` is for `noUncheckedIndexedAccess`.
