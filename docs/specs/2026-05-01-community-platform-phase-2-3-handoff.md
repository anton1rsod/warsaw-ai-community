## Why this prompt is shaped the way it is

- **Same four-files read order** as the Phase 1 handoff (HANDOFF, execution-plan, plan-this-phases-only, CHANGELOG). Plan-Phase-2-and-3-only keeps the read at ~30K tokens instead of ~80K.
- **Phases bundled per execution-plan §10.2** — both phases are read-only, similar shape (parser + page), and tightly correlated risk surface (markdown sanitization affects all of Phase 2 + Phase 3). Bundling avoids re-reading the same sanitization context twice.
- **Outstanding human work explicit** — Anton's §9.10 Vercel Root Directory + Task 1.4 OAuth App are flagged as parallel-track items, not blocking Phase 2/3 implementation. Until Anton finishes them, "Phase 1 live OAuth on a Vercel preview" is unverified end-to-end, but local builds + tests + E2E are fully green.
- **No new architectural decisions expected** — Phase 2 + 3 are well-trodden patterns (markdown parser + table-driven readers + RSC pages). The only architectural surface is the §6.7 sanitization pipeline, which is locked in spec.
- **Closeout deliverables enumerated** — including the next handoff write. Each chat ends by writing its successor's prompt; that's the pattern that makes the 7-chat split viable.
