## 3. Documented adaptations from plan-as-written

### 3.1 Vitest `@/*` alias filesystem-path encoding fix (commit `d3d0892`)

**Trigger:** Task 1.1's first `@/`-aliased import surfaced a latent bug: `vitest.config.ts` aliased `@` via `URL.pathname`, which URL-encodes the literal space in `Warsaw AI Comunity` (the workspace dir name). All prior tests used relative imports, so the bug never fired.

**Fix:** Single-line change — `URL.pathname` → `fileURLToPath(...)`. Committed separately as a prerequisite so Task 1.1's commit could match the plan verbatim.

**Status:** accepted; required for the entire E1 + E2 to function.

### 3.2 `prompts/ask.ts` angle-bracket prose adjustment (commit `9012611`)

**Trigger:** Plan's verbatim impl text uses literal `<excerpts>` three times — twice in prose, once as the structural delimiter. Plan's verbatim test asserts exactly one structural `<excerpts>` open. Both can't simultaneously hold.

**Fix:** Dropped angle brackets from the two prose mentions ("the excerpts block below..." instead of "the <excerpts> block below..."), preserved structural delimiters. Test passes; prompt semantics unchanged (Gemini still sees structural tags below the prose).

**Status:** accepted; **E2 Task 11.1 (injection-cases corpus) must author tests against the landed prompt text at commit `9012611c`, not the plan's quoted text.**

### 3.3 `help/registry.ts` `BotClient` adaptation (commit `4d5719c`)

**Trigger:** Plan's quoted impl forward-declares a simplified `BotClient` interface (`sendMessage(chatId, text, opts?)`) but the existing `src/telegram/client.ts` already exports a fuller `BotClient` (`sendMessage(chatId, threadId, text, parseMode?)` + `sendDirectMessage` + `setReaction`). Plan's NOTE block explicitly anticipates this: "If `telegram/bot.ts` already exports a class, replace the interface..."

**Fix:** Replaced the plan's inline `BotClient` block with `import type { BotClient } from "../telegram/client"`. Phase 8/9/10 command handlers in E2 will get the real surface (4-arg `sendMessage` + DM + reaction methods) instead of the simplified mock.

**Status:** accepted; this is the form the plan's NOTE block instructs.

## 4. Quality-debt backlog (carried forward, not blocking)

The Phase 1 batched code-quality review surfaced 0 Critical, 3 Important, 4 Minor findings against the retrieval/ cluster. Per E1 working agreement (don't deviate from plan code mid-execution; surface as backlog), all are deferred:

### Important
- **[QD-1] `load.ts:35–39`** — post-Zod-parse `embedding_model` and `schema_version` literal-mismatch checks are unreachable given `ManifestSchema`'s `z.literal(...)` constraints. Risk: developer relaxes the Zod literal in the future believing the explicit check still guards them. Recommended fix: remove the explicit checks and add a code comment on the schema cross-referencing this file.
- **[QD-2] `chunk.ts:46`** — guard `if (pos <= 0) pos = end` is unreachable for current constants (`CHUNK_CHARS=1920, OVERLAP_CHARS=200`). If constants are ever tuned to make `OVERLAP > CHUNK`, the guard is silently wrong. Recommended fix: change to `if (pos >= end) pos = end`.
- **[QD-3] `load.test.ts:79–85`** — "stays failed after first failure" test verifies both calls return failure but doesn't prove the cached-failure path was used (vs. two independent failures). Recommended fix: add `expect(a.reason).toBe(b.reason)` or spy on `readFileSync`.

### Minor
- **[QD-4] `chunk.test.ts:39`** — determinism test uses 180-char body (single chunk); doesn't exercise the overlap/window logic. Recommend adding a multi-chunk determinism case.
- **[QD-5] `cite.ts:1`** — module-level `/gi` regex with mutable `lastIndex` is safe today (only used via `replace`) but a known footgun for future `test`/`exec` callers. Recommend constructing inside the function.
- **[QD-6] `schema.ts:8,24`** — `source_lines` tuple permits inverted ranges (`[30, 1]`). Could be caught earlier with `.refine(([s, e]) => s <= e)`. Phase 7's `build-index.ts` is the only writer, but defense is cheap.
- **[QD-7] `load.ts:51`** — `_resetForTests` is exported but unused (tests use `vi.resetModules()`). Recommend marking `@internal` JSDoc.

**Disposition:** address in a single cleanup commit during E3 (after the calibration gate passes) or in 0.1.3. None block E2.

## 5. Architectural decisions logged in E1

- **Branch strategy:** `gbrain-0.1.2-ask-bundle` feature branch, will fast-forward merge to `main` at end of E3 (Task 16.2 tag point). E2 and E3 continue on this branch.
- **Subagent-driven execution:** used `superpowers:subagent-driven-development` for code tasks (Phases 1–6). Phase 0 docs/config tasks done inline. Phase 1 cluster reviewed in batch at end of phase rather than per-task. Single-task phases (2, 5, 6) skipped per-task quality review since the implementer self-review + full-suite-pass cover the contract.
- **Console error logging:** `load.ts` uses `console.error("[gbrain.retrieval.load] ...")` for cold-start failure logging on Vercel. Mirrors the 0.1.1 `gateway.ts` cachedProvider pattern. TypeScript `console.log` rule does not apply to `console.error` for ops-visible errors.
- **`as const satisfies` discoverability invariant:** `COMMAND_REGISTRY` is the type-system enforcement of the spec §4.1/§4.4 item 9 invariant. Removing `description` or `detail` from any entry produces a TS error. This is the load-bearing pattern; don't relax to wider types.
- **Singleton-via-module-scope:** `cachedIndex` (load.ts) and `cachedProvider` (gateway.ts) both use module-level `let` bindings. Test isolation uses `vi.resetModules() + dynamic import()`. Do not refactor to factory/DI patterns just for test ergonomics — the singleton is the architecture.
- **Rate-limit storage:** in-memory `Map` per-region, not distributed. Per spec §4.5 + OQ-8: distributed storage deferred to 0.2.x if a single bursty cross-region attacker appears.
