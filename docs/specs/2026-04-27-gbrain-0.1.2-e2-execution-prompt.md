## Why this prompt is shaped the way it is

- **Five files in priority order, with task-relevant section pointers** — same pattern as the E1 prompt. The prompt steers the chat away from re-reading content that isn't load-bearing (e.g., spec §1–§2 are shipped; §3/§4/§6/§7 are the relevant chunks for E2's work).
- **Explicit "read the LANDED prompts/ask.ts" instruction** — Phase 11's biggest risk is authoring tests against the plan's quoted prompt text and hitting a divergence at the test boundary. Surfacing this once in the prompt avoids 5 round-trips during Task 11.1.
- **Entry conditions are checkable from the chat itself** — `git status`, `pnpm test`, `pnpm typecheck`. If any drift between E1 and E2 occurred, the chat catches it on first run.
- **Constraints repeat the Phase E firewall + the don't-create-new-ADRs rule + the don't-fix-quality-debt rule** — three guardrails that prevent scope creep within E2.
- **No human-in-loop steps in E2** — flagged explicitly so the chat doesn't accidentally try to ask Anton for STAGING_CHAT_IDS (E3 Task 13.1) thinking it's blocking Phase 7.
- **BotClient adaptation note** — re-stated because Phases 8/9/10 implement command handlers against this surface; if the implementer reads only the plan's quoted forward declaration, the integration tests will fail at Phase 8.2's webhook wiring.
- **Acknowledgment-first** — same pattern as E1's prompt; Anton gets a confirm-or-redirect handle before code starts moving.

## What's deliberately NOT in the prompt

- Detailed task-by-task instructions — those live in the plan; duplicating here would rot.
- Phase 12–16 details (E3 scope) — the prompt explicitly carves them out, doesn't enumerate.
- Spec §1, §2, §5, §8–§11 — not load-bearing for E2 work.
- The full ADR-0009 text — Phase 11 needs the test-categories taxonomy summary, not the full ADR. Skim instruction is correct.
- The seven QD- backlog items — closeout doc owns them; the prompt just says "don't fix."
